# REST API 문서 읽는 법
## REST API 
### ? REST API ?
- REST(Representational State Transfer)
- REST API : 웹에서 사용되는 데이터나 자원(Resource)을 HTTP URI로 표현하고, HTTP 프로토콜을 통해 요청과 응답을 정의하는 방식
- 서로 잘 알아볼 수 있도록 작성하는 것이 중요

CRUD
~~~
CREATE : POST /api/user
READ :   POST /api/user
UPDATE : POST /api/user
DELETE : POST /api/user
~~~

### 좋은 REST API 디자인하는 방법
- 레오나르드 리차드슨의 4단계 모델(성숙도 모델)
- 0~ 3단계로 이루어짐
- 2단계까지만 적용해도 좋은 API디자인

### REST 성숙도 모델 - 0단계
- HTTP 프로토콜을 사용하기만 해도 됨

ex.
예약 가능 시간 확인
Request
~~~
POST /appintment HTTP/1.1
[헤더 생략]

{
  "date": "2022-08-10",
  "doctor": "허준"
}
~~~
Response
~~~
HTTP/1.1 200 OK
[헤더 생략]

{
  "slots": [
    {"doctor": "허준", "start": "09:00", "end": "12:00"}
    {"doctor": "허준", "start": "14:00", "end": "16:00"}
  ]
}
~~~
특정 시간에 예약
Request
~~~
POST /appintment HTTP/1.1
[헤더 생략]

{
  "doctor": "허준",
  "start": "14:00",
  "end": "15:00",
  "patient": "김코딩"
}
~~~
Response
~~~
HTTP/1.1 200 OK
[헤더 생략]
~~~
- 모든 요청에서 엔드포인트로 /`appointment`를 사용

### REST 성숙도 모델 - 1단계
- 개별 리소스와의 통신 준수
- 모든 자원은 개별 리소스에 맞는 엔드포인트(Endpoint)를 사용해야 한다는 것
- 요청하고 받은 자원에 대한 정보를 응답으로 전달

ex.
예약 가능 시간 확인
Request
~~~
POST /doctors/허준 HTTP/1.1
[헤더 생략]

{
  "date": "2022-08-10",
}
~~~
Response
~~~
HTTP/1.1 200 OK
[헤더 생략]

{
  "slots": [
    {"doctor": "허준", "start": "09:00", "end": "12:00"}
    {"doctor": "허준", "start": "14:00", "end": "16:00"}
  ]
}
~~~
특정 시간에 예약
Request
~~~
POST /slots HTTP/1.1
[헤더 생략]

{
  "patient": "김코딩"
}
~~~
Response
~~~
HTTP/1.1 200 OK
[헤더 생략]
{
  "appointment": {
    "slot": {"id": "123", "doctor"": "허준", ...},
    "patient": "김코딩"
  }
}
~~~
- 요청하는 리소스가 무엇인지에 따라 각기 다른 엔드포인트로 구분하여 사용
- 예약 가능한 시간 확인이라는 요청의 응답으로 받게 되는 자원(리소스)은 허준이라는 의사의 예약 가능한 시간대이므로 요청 시 `/doctors/허준`이라는 엔드포인트를 사용
- 특정 시간에 예약하게 되면, 실제 slot이라는 리소스의 123이라는 id를 가진 리소스가 변경되므로 하단의 특정 시간에 예약이라는 요청에서는 `/slots/123`으로 실제 변경되는 리소스를 엔드포인트로 사용
- 적절한 엔드포인트를 작성하는 것이 중요

- 특정 시간 예약이 불가능 할 때

Response
~~~
HTTP/1.1 200 Conflict
[헤더 생략]
{
  "appointmentFailure": {
    "slot": {"id": "123", "doctor"": "허준", ...},
    "patient": "김코딩",
    "reason": "해당 시간은 이미 예약되어 있습니다"
  }
}
~~~
### REST 성숙도 모델 - 2단계
- CRUD에 맞게 적절한 HTTP 메소드를 사용하는 것에 중점

ex.
예약 가능 시간 확인
Request
~~~
GET /doctors/허준/slot?date=2022-08-10 HTTP/1.1
[헤더 생략]

~~~
Response
~~~
HTTP/1.1 200 OK
[헤더 생략]

{
  "slots": [
    {"id": "123","doctor": "허준",...}
    {"id": "124","doctor": "허준",...}
  ]
}
~~~
특정 시간에 예약
Request
~~~
POST /slots HTTP/1.1
[헤더 생략]

{
  "patient": "김코딩"
}
~~~
Response
~~~
HTTP/1.1 201 Created
Location: slots/123/appointment
[헤더 생략]
{
  "appointment": {
    "slot": {"id": "123", "doctor"": "허준", ...},
    "patient": "김코딩"
  }
}
~~~
메소드 Rules
- `GET` : 서버의 데이터를 변화시키지 않는 요청에 사용
- `POST` : 요청마다 새로운 리소스를 생성
- `PUT` : 요청마다 같은 리소스를 반환
  - 요청마다 같은 리소스를 반환하는 특징을 멱등(idempotent)하다고 함
  - 멱등성을 가지는 메소드 PUT과 그렇지 않은 POST는 구분
- `PUT(교체)` 과 `PATCH(수정)` 도 구분
### REST 성숙도 모델 - 3단계
- HATEOAS(Hypertext As The Engine Of Application State)로 표현되는 하이퍼미디어 컨트롤 적용
- 응답에는 리소스의 URI를 포함한 링크 요소를 삽입하여 작성

ex.
예약 가능 시간 확인
Request
~~~
GET /doctors/허준/slot?date=2022-08-10 HTTP/1.1
[헤더 생략]

~~~
Response
~~~
HTTP/1.1 200 OK
[헤더 생략]

{
  "slots": [
    {"id": "123","doctor": "허준",..},...
  ],
  "links" :{
    "appointment" : {
      "href" : "http://localhost:8000/slots/123",
      "method" : "POST"
    }
  }
}
~~~
특정 시간에 예약
Request
~~~
POST /slots HTTP/1.1
[헤더 생략]

{
  "patient": "김코딩"
}
~~~
Response
~~~
HTTP/1.1 201 Created
Location: slots/123/appointment
[헤더 생략]
{
  "appointment": {
    "slot": {"id": "123", "doctor"": "허준", ...},
    "patient": "김코딩"
  },
  "links" : {
    "self" : {
      "href" : "http://localhost:8000/slots/123",
      "method" : "GET"
    },
    "cancel" : {
      "href" : "http://localhost:8000/slots/123/cancel",
      "method" : "DELETE"
    }
  }
}
~~~
- 응답 내에 새로운 링크를 넣어 새로운 기능에 접근할 수 있도록 하는 것이 3단계의 중요 포인트
